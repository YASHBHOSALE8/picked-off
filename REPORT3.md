## Stream pool

- **200 streams = 40 per level × 5 levels**, at the final §1.5 regime (`lambda_j=0.2, p_jump=0.2, lambda_a=4.0, delta0=8.0`, α per level table). 4.9 MB total, ~25 KB each; one file fetched per round, lazily.
- Every stream is generated by the frozen `picked_off.generator` with §6.3-5 timestamp hygiene and the Q5 V>0 path check (deterministic seed re-roll on rejection, mirroring the gate runner). Jumps + arrivals only — the player supplies the quotes.
- Round start picks uniformly from the level's pool (`Math.random`, UI-level only — engines stay RNG-free per §0-3). The `stream_id` (= `L{level}-{seed}`) is shown in the replay HUD and embedded in the session export, so any session is exactly reproducible.

## DESIGN.md §7 / brief deviations (explicit)

1. **Live HUD shows spread captured only — not AS/IC ticking.** The brief asked for a live SC/AS/IC decomposition; AS and IC are functions of the hidden V, so showing them live would leak the fair value through the PnL display and break §1.4's censoring (a player could read V off the at-fill AS). DESIGN.md is source of truth → the HUD shows clock, position, live SC, and "AS / IC revealed at close". The score screen has the full identity-exact decomposition seconds later.
2. **Q3 resolved as specified by the brief**: fills are side-colored live (buy lifts = cyan, sell hits = amber). Trader type is *never* used in live rendering (censoring discipline); red pick-off markings appear only in the replay. Revisit in playtest per the brief.
3. **New file location**: `web/scripts/build_stream_pool.py` is not in the §7.1 layout tree (additive; sim/ itself untouched).
4. **Environment**: node/npm did not exist on this machine; Node 22 LTS was installed **user-space only** (`~/.local/node22`, official tarball, no sudo, removable with `rm -rf ~/.local/node22`). Nothing system-level changed.

## Known UI rough edges (listed, not fixed — playtesting next)

1. The live price window is a fixed 90-tick span auto-centered on your quote mid; if V drifts far and you chase it fast, fills can briefly flash near the canvas edge before the view eases over (cosmetic; your own quotes are always in view).
2. Replay decline markers can overlap into clutter at tight spreads with busy flow (L4/L5); no declutter/zoom yet.
3. Quote dragging is pointer-only — no keyboard accessibility for the lines, no nudge buttons.
4. The replay re-renders React state per animation frame while playing (fine on tested hardware, unprofiled on low-end phones).
5. No persistence (no best-score memory, no settings) — localStorage is permitted by §7.3 but unused.
6. Level-select "how it works" link is a stub (the README is the doc); no in-game tutorial beyond the level copy and arm-screen hint.
7. The arm screen's default quotes are V0±3 regardless of level; a per-level default might teach better.
8. Session-export caveat: live-round `u_accept` margins are certified for hygiene/V>0 but cannot be pre-certified against *player* quotes (unknowable), so re-running an exported session through the **Python** engine could in principle flip one comparison by a 1-ulp `exp()` difference. Within the web engine, replays are bit-exact. (Same situation as the Python bot harness; theoretical probability ~10⁻⁹ per fill.)
9. Mobile check was code-level only (viewport meta, `touch-action: none`, 36 px grab slop, ≥42 px buttons, wrap at 390 px) — no real-device pass; that's what playtesting is for.
